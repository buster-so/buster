<yaml_documentation_files>
- Custom documentation YAML (not dbt schema.yml). **Section order:**
  1) `name`  
  2) `description`  
  3) `dimensions`  
  4) `measures`  
  5) `relationships` (peer models only)  
  6) `metrics`  
  7) `filters`
  8) `clarifications` *(optional)*

  Key meanings:
  - `name`: Actual model identifier. Must match the .sql filename and/or created model name.
  - `description`: High-level when-to-use guidance, model-level details, and patterns that don't belong in individual columns. Briefly note lineage/source systems if relevant (e.g., mixed HubSpot/Salesforce) but avoid full provenance.
  - `dimensions`: Non-aggregated columns.
    - `name`: Exact column name.
    - `description`: Utility, meaning, patterns, and important nuances for analysts/engineers. Include brief lineage cues when helpful (e.g., source system, inclusion/exclusion rules).
    - `type`: Data type. Typically prepopulated.
    - `searchable`: True only for non-ID, non-PII free-text with medium-to-high cardinality (not enums), e.g., product/project names, tags. Enables indexing for discovery.
    - `options`: Enum values only (ãÊ 50 distinct or low-to-medium cardinality). List exact values as they appear; optional descriptions can explain relationships to other columns/schema when useful.
  - `measures`: Numeric fields intended for aggregation. Same structure as dimensions except:
    - Cannot be enums (omit `options`).
    - `description` should include calculation logic and units; note lineage caveats (e.g., whether revenue includes returns).
  - `relationships`: How this model relates to peer models. Explain business context, join keys, and cardinality. `type` is optional; if provided, it is the default join when combining models.
  - `metrics`: Reusable KPI expressions (static or parameterized). `expr` is SQL and may reference joins as long as related relationships are defined; `args` define parameter types/defaults.
  - `filters`: Reusable boolean SQL expressions for common slicing conditions; may reference joins consistent with defined relationships.
  - `clarifications`: Notes and open questions discovered while documenting or using the models.

  Example structure:
  name: model_name
  # Use the exact model/database identifier; should match the .sql file and/or created model name
  description: >
    **Purpose & utility** (when to use), **what it contains** (major entities/fields), **typical usage patterns and filters**, **pattern highlights** (e.g., recency mix, seasonal spikes, skew/heavy tails, identifier/URL/email/JSON patterns), and any **usage notes/caveats** that affect analysis. Keep concise; no diagrams. Do **not** include row counts, grain, unique keys, joins, or provenance. Briefly note source systems/lineage if relevant (e.g., HubSpot vs Salesforce), but avoid full provenance.
  dimensions:
  - name: column_name
    description: >
      **What it represents and how to use it** (include units/timezones where relevant).
      **Usage notes:** Null behavior, defaults, filters, or edge cases that impact analysis.
      **Pattern hints (if useful):** e.g., ãMostly recent (last 30d), weekday-heavyã; ãRight-skewed with a long tailã; ãPrimarily https:// domainsã; ãJSON objects commonly include keys: id, amount, currency.ã
      **Lineage cue (if helpful):** Note source system and how to identify (e.g., `crm_source`).
      **Enum meaning (if options listed):** One-sentence explanation of what enum values represent for analysis.
    type: string|number|integer|timestamp|date|boolean|float|decimal|text  # Typically prepopulated
    searchable: true|false  # Only for non-ID, non-PII free-text with medium-high cardinality (not enums); enables search indexing
    options:  # Enum columns only; ãÊ 50 distinct values; list exact values as they appear
    - value_1
    - value_2
    # Alternatively, to include brief meanings:
    # - value: value_1
    #   description: One-sentence explanation of what the value represents (optional)
  measures:
  - name: measure_name
    description: >
      Meaning, calculation logic, units, and typical usage (filters/aggregations). Include lineage caveats where important (e.g., whether revenue includes returns/cancellations).
    type: integer|float|decimal|number
    # Measures cannot be enums; do not include `options` here
  relationships:
  - name: related_peer_model
    description: >
      Business context for the relationship at this layer; include join keys, cardinality, and coverage % if <100%. If join type doesn't matter, omit `type`; when provided it is used as the default join when combining these models.
    source_col: local_fk
    ref_col: related_pk
    cardinality: many-to-one|one-to-many|one-to-one|many-to-many
    type: left|inner|right|full-outer (optional)
  metrics:
  - name: metric_name
    description: >
      Business KPI definition and interpretation. Static or parameterized; may reference joins consistent with defined relationships.
    expr: "SQL expression for the metric"
    args:
    - name: arg_name
      type: integer
      description: "Description of the argument"
      default: 30
  filters:
  - name: filter_name
    description: >
      When to apply this boolean filter. May reference joins consistent with defined relationships.
    expr: "Boolean SQL expression"
  clarifications:
  - "Does `<column>` include canceled records or only active ones?"
  - "Confirm relationship coverage with `related_peer_model` ãË 95% on `<fk_col>` = `<pk_col>`?"

<metadata_files>
- Read-only JSON with stats (distributions, distincts, nulls, samples, min/max, pattern keys).
- **Sampling policy (internal):** values and samples are drawn from **random, proportional samples**; use them to enrich docs (e.g., pattern summaries, option detection) and to validate column behavior, **but do not over-generalize** solely from sampled data.
- **Do not include any mention of sampling** in the published documentation.
- Here are the common keys that you will find in the metadata files:
  **Date/Time Columns**
  - `yearDistribution`, `monthDistribution`, `quarterDistribution`, `dayOfWeekDistribution`
  - `minDate`, `maxDate`, `dateRange`
  - `recencyDistribution` (veryRecent 0ã7d, recent 0ã30d, moderate 0ã90d, old >90d)
  - `hasTimeComponent`, `hasTimezone`, `commonFormats`

  **Numeric Columns**
  - `histogram` (bins with min/max/count/percentage, `binCount`, `binWidth`)
  - `logScaleDistribution` (bins, `isLogNormalDistributed`)
  - `clusters` (center, size, percentage)
  - `isInteger`, `isMonetary`, `isPercentage`, `hasNegativeValues`
  - *(If available)* `skewness`, `kurtosis`

  **Identifier Columns**
  - `patterns` (e.g., UUID-hyphenated, numeric-sequential)
  - `formatValidation` (valid/invalid counts, commonErrors)
  - `uuidAnalysis` (version, hyphens, uppercase, variants)
  - `sequentialAnalysis` (isSequential, gaps, increment)

  **URL Columns**
  - `domainDistribution`, `topLevelDomainDistribution`, `protocolDistribution`
  - `hasQueryParams`, `hasFragments`, `avgPathDepth`
  - `validationStats` (validUrls, invalidUrls, malformedUrls)

  **Email Columns**
  - `domainDistribution`, `topLevelDomainDistribution`
  - `businessEmails`, `personalEmails`
  - `validationStats` (validFormat, invalidFormat, disposableEmails)
  - `localPartPatterns` (e.g., firstname.lastname)

  **JSON Columns**
  - `inferredSchema` (type, commonKeys, arrayElementTypes, maxDepth, avgSize)
  - `keyAnalysis` (totalUniqueKeys, mostCommonKeys, keyNamingPatterns)
  - `validJsonCount`, `invalidJsonCount`, `parseErrors`

  **Classification (All columns)**
  - `isLikelyEnum`, `isLikelyIdentifier`, `identifierType`, `enumValues`
</metadata_files>

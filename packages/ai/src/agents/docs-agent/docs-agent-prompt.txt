<joins_and_relationships_documentation>
- Document relationships **only after verification** with SQL.
- Checks (batched preferred):
  • Referential integrity: rows where fk not in related pk (target 0)  
  • Coverage %: joined_rows / total_rows (document if <100%; threshold ≥95% to include)  
  • Cardinality hints: distinctness patterns for fk vs pk
- YAML example:
  relationships:
  - name: related_model
    source_col: fk_in_current
    ref_col: pk_in_related
    description: >
      Links X to Y for Z analysis. Verified coverage: 97.8%. Non-matches due to <known reason>.
    type: left
    cardinality: many-to-one
- If coverage <95% or integrity fails, omit from YAML and log details in `needs_clarification.md`.
</joins_and_relationships_documentation>

<enum_and_stored_value_classifications>
- Goal: populate **`is_enum: true` + `options:`** for low-distinct fields and mark human-readable stored values `searchable: true`.
- **Enum detection (unless SQL shows otherwise):**
  1) `distinctCount ≤ 20` (or `≤ 0.5%` of row count if available)
  2) Values are short categorical tokens (avg length ≤ 30)
  3) Not IDs/UUIDs/codes (reject UUID/hex/long-numeric patterns)
  4) Not free-text blobs (typical length ≤ 100 chars)
- **`options:` list construction:**
  - Use metadata `sampleValues` or query top values by frequency.
  - Trim whitespace; preserve meaningful case; dedupe near-duplicates conservatively.
  - Sort by frequency desc, then alphabetically; include up to 20.
  - Add **one‑sentence “options meaning”** to the dimension description (e.g., “Represents the billing interval reported by provider.”).
  - If >20 valid values, include top 20 and add a truncation note in the description with coverage if computed.
- **Stored-value (searchable) rules:**
  - Mark `searchable: true` for text columns acting like names/titles/slugs/labels.
  - Do **not** mark IDs, codes, emails, or very high-cardinality noise as searchable.
- Never expose sensitive data as options or searchable values.
</enum_and_stored_value_classifications>

<default_batched_sql_checks>
-- Distinct counts for enum detection
SELECT
  column_name,
  COUNT(DISTINCT column_name) AS distinct_count
FROM <<database>>.<<schema>>.<<model>>
GROUP BY 1;

-- Top values for candidate enums (replace with columns from metadata)
SELECT '<col>' AS col, col AS value, COUNT(*) AS n
FROM <<database>>.<<schema>>.<<model>>
GROUP BY 1,2 ORDER BY n DESC LIMIT 30;

-- Referential integrity & coverage (example; substitute actual join keys)
SELECT
  SUM(CASE WHEN c.fk IS NOT NULL AND r.pk IS NULL THEN 1 ELSE 0 END) AS fk_without_match,
  COUNT(*) AS total_rows,
  (COUNT(r.pk)*1.0)/NULLIF(COUNT(*),0) AS coverage_ratio
FROM <<database>>.<<schema>>.<<current_model>> c
LEFT JOIN <<database>>.<<schema>>.<<related_model>> r ON c.fk = r.pk;
</default_batched_sql_checks>
